# Claude Code 行動原則 — note-auto-revenue プロジェクト

## 問題フラクタル展開プロセス

システムの実行中に何らかの問題・制約・エラーに直面したとき、
即座に諦めたり停止したりしないこと。以下の3ステップで必ず解決策を探す。

### STEP 1: 問題を1つ上のレイヤーで捉え直す
- 「なぜこれが問題になっているか？」を1段抽象化する
- 表面的な技術問題を、本質的な制約として言語化する

### STEP 2: そのレイヤーでの解決策を3つ列挙する
- 回避策・代替手段・迂回路を探す
- コストが発生するなら、そのコストを収益で賄う方法を考える

### STEP 3: さらに解決できなければ、もう1段上げて繰り返す
- 最終的に「何を達成すれば根本解決になるか」まで遡る

### フラクタル展開の例

| 問題カテゴリ | Lv1（表面） | Lv2（1段上） | Lv3（根本） | 解決策 |
|---|---|---|---|---|
| 技術系 | APIレート制限 | 1アカウントの処理量上限 | スケールに複数アカウントor上位プランが必要 | 上位プランのコストを収益目標に加算して先に達成する |
| リソース系 | メモリ不足 | 無料枠では処理が重すぎる | 有料サーバーへの移行コストが必要 | 月収がサーバー代を超えるまで処理を軽量化して耐える |
| 収益系 | 記事が売れない | 読者に届いていないorニーズとズレ | 市場調査と配信チャネルの見直しが必要 | scout_agentのキーワードを変更し配信先を追加する |

### 問題報告フォーマット

```
【問題報告】
発生した問題: （具体的に）
フラクタル展開:
  Lv1（表面の問題）:
  Lv2（1段上の本質）:
  Lv3（根本の制約）:
提案する解決策（3案）:
  案A: （今すぐできる回避策）
  案B: （収益を使った中期解決）
  案C: （根本解決の長期プラン）
推奨案: （AかBかCと、その理由）
私への確認事項: （必要な場合のみ）
```

### 重要な原則

- **「できません」で終わらない。必ず次の行動を提案する**
- 問題を収益機会として捉え直す視点を持つ（コストが必要→そのコストを出す収益目標を設定する）
- 解決に時間がかかる場合は、その間の暫定対策も同時に提案する
- 人間（オーナー）の判断が必要な場合だけ報告する。自力で解決できる問題は自己判断で対処する

## ビジネス現実制約（新機能実装前に必ず確認）

### 原則: 集客 → 信頼 → 収益化 の順番を絶対に守る

どんな収益モデルでも以下の順序を飛ばしてはいけない:
1. 無料コンテンツで読者を集める（最低30日間）
2. 無料コンテンツで信頼を積む（最低10〜20本）
3. その後はじめて有料・課金・広告を導入する

### プラットフォーム別の現実制約

**note**: フォロワー0・記事0で有料記事を出しても売れない。最初の1ヶ月は無料記事のみ。有料は無料記事10本以上になってから。

**アフィリエイト**: 記事0本では審査が通らない。記事10本以上・更新頻度・内容の質が必要。広告タグ設置は審査通過後のみ意味がある。

**Gumroad**: 集客元（note/ブログ）を先に育ててから誘導。

**GitHub Pages**: 公開直後はインデックスされない（2〜4週間）。記事30本以上必要。

### 新機能実装前の必須チェック

実装前に以下を自問し、答えを提示すること:

- Q1. この機能が動くための「前提条件」は何か？
- Q2. その前提条件は今すでに満たされているか？満たされていなければ先に何をすべきか？
- Q3. 今の状態でこれを動かしたとき、実際に収益につながるか？
- Q4. 「技術的に動く」と「ビジネスとして機能する」の両方を満たしているか？

Q2またはQ3がNOなら実装を止めて、先にすべきことを提案する。

### フェーズ定義

| フェーズ | 条件 | やること | 禁止事項 |
|---|---|---|---|
| フェーズ0（種まき） | 記事<10本 または フォロワー<50人 | 無料記事の自動生成・投稿のみ | 有料記事・広告・課金 |
| フェーズ1（芽吹き） | 無料記事10本以上 かつ フォロワー50人以上 | 有料記事・アフィリエイト申請 | 有料:無料=3:7を維持 |
| フェーズ2（成長） | 月収>3,000円 かつ 3ヶ月連続収益 | 価格最適化・新事業検討 | 土台を崩さず拡張 |
| フェーズ3（拡大） | 月収>10,000円 かつ コア安定 | 帝国アーキテクチャ全起動 | — |

### オーナー確認が必須な判断

自動実行せず必ず確認する:
- 初めてのプラットフォームへの登録・申請
- 無料→有料への切り替え
- 新しいフェーズへの移行
- 月コストが前月比+50%超える変更
- 収益が出ている事業の仕様変更

確認フォーマット:「【確認が必要】〇〇をしようとしています。現在の状態: [記事数・フォロワー数・月収] / 実行する理由: / リスク: / 実行してよいですか？」

## プロジェクト概要

**自律収益帝国** — Claude API + GitHub Actions で複数事業を自動運営する収益システム。

### 主要ファイル

| ファイル | 役割 |
|---|---|
| `empire/empire_main.py` | 帝国全体のオーケストレーター（daily/weekly/monthly） |
| `empire/ceo_agent.py` | 戦略的意思決定（2層構造: Pythonルール + Claude判断） |
| `empire/scout_agent.py` | 新市場スキャン（3条件ゲート付き） |
| `empire/launcher_agent.py` | 新事業自動セットアップ（承認制） |
| `empire/portfolio.yaml` | 全事業の中央ステート |
| `empire/playbooks/` | 4種の事業テンプレート |
| `config.yaml` | Zenn事業の設定（ユーザー編集用） |

### 事業プレイブック

| ID | 事業種別 | スケールトリガー |
|---|---|---|
| `note_magazine` | note有料マガジン | 購読者50人超 |
| `gumroad_product` | Gumroadデジタル商品 | 月10本以上販売 |
| `affiliate_seo` | アフィリエイト×SEO記事 | 月間PV 1,000超 |
| `api_service` | 簡易APIサービス（RapidAPI） | 月100リクエスト超 |

### 意思決定ルール（CEO判断基準）

**スケールアップ**
- 前月比+20%収益増 → 予算1.5倍
- 3ヶ月連続成長 → 最優先事業に昇格
- ROI >200% → 水平展開を検討

**スケールダウン/停止**
- ROI <50% が2ヶ月継続 → 予算半減
- ROI <50% が3ヶ月継続 → 停止
- コスト>収益 が2ヶ月継続 → 即時停止

**新事業展開の3条件（全て揃ったとき）**
1. 収益プールの available_budget > ¥10,000
2. アクティブ事業数 < 5件
3. scout_agent スコア ≥ 35点

**収益再投資の優先順位**
1. 運用コスト（Claude API代等）
2. 成長投資（既存事業の予算増）
3. 新事業準備金（¥10,000プール）
4. オーナー利益（残額）

### 収益目標

| 期間 | 目標 |
|---|---|
| 3ヶ月 | ¥30,000/月 |
| 6ヶ月 | ¥100,000/月 |
| 12ヶ月 | ¥300,000/月 |
| 最大事業数 | 10件 |

### 安全装置（変更禁止）

- 月次APIコスト上限を超えたら全処理停止
- 新事業の起動には必ずオーナー承認（`pending_human_approval` → 手動でActiveに）
- 事業停止には24時間の猶予（`pending_kill` → 翌日実行）
- 新事業は1日1件まで
