## ログイン壁という「全の欠損」

副業の自動化を試みた人間の9割は、同じ場所で止まる。

ログイン画面だ。

スクレイピングのコードを書いた。Playwrightのスクリプトも動いた。Claude Codeからブラウザを操作する仕組みも理解した。それでも、実際に業務として使えるレベルには届かない。なぜか。セッションが切れるたびに、自動化が崩壊するからだ。

これはスキルの問題ではない。構造の問題だ。

自動化スクリプトは「白紙のブラウザ」として起動する。認証情報を持たない。あなたが毎日使っているChromeの「ログイン済みの状態」を引き継げない。だから毎回、ログイン処理を自分でコードに書かなければならない。2段階認証が挟まれば詰む。企業のシングルサインオンが入れば詰む。

副業でよく使うツール——freee、マネーフォワード、クラウドワークス、Notion、各種APIのダッシュボード——のほとんどは、このログイン壁の内側にある。

そしてここに、見落とされている構造的な解法がある。

## 「新しいブラウザを起動する」という発想が間違っている

PlaywrightもCypressも、デフォルトの発想は「新しいブラウザを立ち上げる」だ。

でも考えてほしい。あなたは今、Chromeを開いている。そのChromeには、すでにfreeeにもGoogleにもNotionにもログインしている。セッションクッキーが全部生きている。

なぜ、そのブラウザをそのまま使わないのか。

`playwright-cli attach` という機能がある。正確には、PlaywrightのCDPブリッジを使って「すでに起動しているChromeにアタッチする」手法だ。新しいブラウザを起動するのではなく、今あなたが使っているChromeのウィンドウに、スクリプトから接続する。

これが何を意味するか。

ログイン処理が一切不要になる。あなたがログインしているなら、スクリプトもログインしている。セッション切れを心配する必要がない。2段階認証のコードを毎回入力するコードを書かなくていい。

「全」（完全な自動化）と「1」（今日の一作業）のあいだにあった壁が、一つ消える。

## 仕組みの核心：CDPとリモートデバッグポート

少しだけ構造を説明する。知っておくと、あとで詰まらない。

ChromeにはCDP（Chrome DevTools Protocol）というインターフェースが内蔵されている。これはブラウザの内部状態を外部から操作するためのプロトコルだ。Playwrightはこれを使ってブラウザを制御している。

通常のPlaywrightは、新しいChromeプロセスを起動してCDPで接続する。でも、リモートデバッグポートを開いた状態で起動済みのChromeがあれば、そこに「外から」接続できる。

これが `chromium.connectOverCDP()` というAPIで実現される。

つまり構造はシンプルだ。

1. Chromeをデバッグポート付きで起動する（一度だけ）
2. Playwrightスクリプトからそのポートに接続する
3. あなたのセッションのまま、なんでもできる

---

ここからが本題だ。

## 実装：Claude Codeから動かすまでの全手順

### Step 1：デバッグポート付きでChromeを起動する

Macなら以下のコマンドをターミナルから実行する。

```bash
/Applications/Google\ Chrome.app/Contents/MacOS/Google\ Chrome \
  --remote-debugging-port=9222 \
  --user-data-dir=/tmp/chrome-debug-session
```

`--user-data-dir` を別パスにしているのは、既存のChromeプロセスと競合しないためだ。ここで指定したディレクトリにログイン情報も保存されるので、2回目以降は自動でセッションが復元される。

Windowsなら：

```bash
"C:\Program Files\Google\Chrome\Application\chrome.exe" ^
  --remote-debugging-port=9222 ^
  --user-data-dir=C:\chrome-debug-session
```

このChromeウィンドウで、副業で使うサービスに手動でログインしておく。freeeでもマネーフォワードでもいい。一度ログインすれば、次回からこのディレクトリにセッションが残る。

### Step 2：Playwrightからアタッチするコードを書く

```python
import asyncio
from playwright.async_api import async_playwright

async def main():
    async with async_playwright() as p:
        # 起動済みChromeに接続
        browser = await p.chromium.connect_over_cdp("http://localhost:9222")
        
        # 既存のコンテキスト（セッション）を取得
        context = browser.contexts[0]
        
        # 既存のページを取得するか、新しいタブを開く
        page = context.pages[0] if context.pages else await context.new_page()
        
        # ここからはログイン済みの状態で操作できる
        await page.goto("https://app.freee.co.jp/reports/trial_balance")
        
        # ページのタイトルを確認
        print(await page.title())
        
        await browser.close()

asyncio.run(main())
```

`connect_over_cdp` の一行が全てだ。これだけで、今あなたがログインしているChromeのセッションをそのまま引き継いだ状態でスクリプトが動く。

### Step 3：Claude Codeに渡す

Claude Codeのプロジェクトに上記のスクリプトを置く。あとはClaude Codeに自然言語で指示を出すだけだ。

例えばこういう指示が機能する。

```
freeeの試算表を開いて、今月の経費合計を取得して、
Notionの家計管理DBに記録してください。
```

Claude Codeはこのスクリプトを読んで、必要なPlaywrightの操作コードを生成・実行する。ログインは不要。セッションはすでにある。

実際にこれをfreeeの月次経費集計で使い始めて、毎月だいたい2〜3時間かかっていた手作業が20分以下になった。年間に換算すると、30時間以上の圧縮だ。時給を計算すれば数字はわかる。

### Step 4：副業での実用的なユースケース

この仕組みが特に刺さる場面を3つ挙げる。

**freeeの経費入力自動化**
領収書データをCSVで持っていれば、freeeにログイン済みのChromeを通して自動入力できる。freeeのAPIではカバーできない細かいUI操作も、Playwrightなら対応できる。

**クラウドワークスの案件管理**
新着案件をチェックして、条件に合うものをスプレッドシートに書き出す。毎日30分かけてブラウザを手動で見ていたなら、それがゼロになる。

**マネーフォワードのレポート取得**
マネーフォワードのPDF出力は手動操作が必要な場面がある。Playwrightでボタンを押して、ファイルをダウンロードして、Notionに添付するまでを全自動にできる。

### Step 5：セッションの永続化と運用

`--user-data-dir` を固定のパスにしておくと、Chrome起動のたびにログイン状態が復元される。

```bash
# Macなら ~/Library 以下に置くと管理しやすい
--user-data-dir=$HOME/Library/Application\ Support/ChromeDebugSession
```

このディレクトリをバックアップしておけば、PCを変えても移行できる。セッションクッキーごとコピーできるので、ログインし直しは不要だ。

ただし一点だけ注意がある。このディレクトリには認証情報が含まれる。Gitには絶対に含めないこと。`.gitignore` に必ず追加する。

```
# .gitignore
chrome-debug-session/
ChromeDebugSession/
```

### Claude Codeのプロンプト設計

最後に、Claude Codeに渡すプロンプトの書き方だけ触れておく。

抽象的な指示より、「どの要素を操作するか」を具体的に書いたほうが精度が上がる。

悪い例：「freeeの経費を入力して」

良い例：「freeeのアプリを開いて、左メニューの『取引』→『取引の一覧・登録』から手動入力フォームを開き、以下のデータを1件ずつ入力してください。日付：2024-01-15、金額：3500、勘定科目：消耗品費、摘要：文房具」

要素が明確なほど、Claude Codeが生成するPlaywrightコードの精度が上がる。スクリプトが壊れたときのデバッグも速くなる。

## ログイン壁は「壁」ではなかった

最初に戻る。

副業の自動化が止まる「ログイン壁」は、技術的な限界ではなかった。「新しいブラウザを起動する」という、デフォルトの発想から来る見かけ上の壁だった。

すでにあるChromeのセッションをそのまま使う。それだけのことで、壁は消える。

自動化の「全」——すべての手作業がなくなった理想状態——は、遠い未来の話ではない。今あなたがログインしているChromeの中に、すでに構造として内包されている。

あとは一本のコードで、そこに接続するだけだ。
