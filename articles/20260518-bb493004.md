## 5つのLLMが「全員バカ」になる理由

正直、半年くらい気づかなかった。

Claude Codeで書きかけたコードをCursorで引き継ぐとき、毎回同じ説明をしていた。「このプロジェクトはNext.jsで、バックエンドはSupabaseで、命名規則はキャメルケースで——」。それを5つのツールそれぞれに、毎回、最初から。

作業の本題に入る前に10分が消える。週5日作業すれば50分。月20日なら200分。年間で計算すると40時間を超えた。副業で40時間あれば、コンテンツ10本は余裕で書ける。

これは効率化の話ではない。構造の話だ。

## LLMが「文脈を忘れる」のは欠陥じゃない

よく「AIは記憶がない」と言われる。でも正確じゃない。正確には「記憶の場所があなたの外にない」だ。

人間でも同じことが起きている。新しい職場に転職したとき、前の会社での5年分のコンテキストは誰も知らない。だから最初の3ヶ月はほぼ「自己紹介とオンボーディング」に消える。LLMはそれが毎回起きているだけだ。

問題はLLMにあるんじゃなくて、コンテキストの置き場所を外部化していないことにある。

## 5LLMを横断する「1ファイル記憶術」の構造

ここで一つの問いを立てる。

「5人の優秀なアシスタントが全員同じ文脈を共有しているとしたら、あなたは何を毎朝説明しなくていいか」

答えが出たら、それがそのままファイルの中身になる。

私が実際に使っているのは `CONTEXT.md` という1ファイルだ。プロジェクトのルートに置く。GitHubにもpushする。Claude Code、Cursor、Gemini、ChatGPT、Codexの5つ、全部このファイルを会話の冒頭に貼り付けるか、参照させる。

仕組みはシンプルすぎるくらいシンプルだ。だからこそ続く。

## CONTEXT.mdの実際の中身（全文公開）

私が副業プロジェクトで使っている `CONTEXT.md` の構成はこうなっている。

```
# PROJECT CONTEXT

## 基本情報
- プロジェクト名: [名前]
- 技術スタック: Next.js 14 / Supabase / Tailwind CSS
- デプロイ先: Vercel
- リポジトリ: github.com/xxx/yyy

## 命名規則
- コンポーネント: PascalCase
- 関数・変数: camelCase
- DBテーブル: snake_case
- ブランチ: feature/xxx

## 私のコーディングスタイル
- コメントは日本語で書く
- 型はできるだけ明示する（any禁止）
- エラーハンドリングは必ず書く
- 1ファイル200行以内を目安にする

## 今週の作業コンテキスト
- 現在取り組んでいる機能: [記入]
- 直近の決定事項: [記入]
- 次のマイルストーン: [記入]

## NGパターン（過去に失敗したこと）
- useEffectの依存配列を省略すると後で必ず問題が出た
- Supabaseのrow level securityを後付けすると工数が3倍になった

## 私の副業の制約
- 作業時間: 平日夜2時間、土日4時間
- 優先順位: 動くものを早く → 後でリファクタ
```

これを最初に貼り付けるだけで、AIへの「自己紹介タイム」が完全に消える。

## 5つのツールへの渡し方、それぞれの正解

ツールごとに「渡し方」が微妙に違う。ここで間違えると、貼っているのに無視される状態になる。

**Claude Code（claude.ai / API）**

プロジェクトルートに `CLAUDE.md` という名前で置くと、Claude Codeが自動的に参照する。ファイル名がそのまま命令になる。`CONTEXT.md` を作ったなら、`CLAUDE.md` にシンボリックリンクを張るか、内容をコピーして使う。Claude Codeはこれを「プロジェクト知識」として扱う。

**Cursor**

`.cursorrules` というファイルをルートに置く。これがCursorの「常時記憶」になる。書き方はMarkdownで自由。私は `CONTEXT.md` の内容をそのままコピーして `.cursorrules` に入れている。Cursorはこれをすべてのチャットの前提として読む。

注意点が一つある。`.cursorrules` が長すぎると（3000字超あたりから）、モデルによっては後半を読み飛ばすことがある。重要な情報は冒頭200字以内に入れる。

**ChatGPT（GPTs / カスタム指示）**

「カスタム指示」の「AIに覚えておいてほしいこと」欄に `CONTEXT.md` の要点を入れる。ただし文字数制限が1500字前後なので、全文は入らない。ここには「技術スタック・命名規則・私のスタイル」の三つだけに絞る。毎回の会話で貼るコストより、カスタム指示に入れておく方が確実だ。

**Gemini（Google AI Studio / Gemini Advanced）**

Geminiにはカスタム指示の概念がまだ弱い。現状は「System Prompt」欄に直接貼る方法が一番確実だ。Google AI Studioを使っている場合、System Promptを保存したプロンプトセットとして残せる。私は「副業プロジェクト用」という名前でセーブしている。

**GitHub Copilot / Codex**

`.github/copilot-instructions.md` に書く。これはCopilotがリポジトリ全体のコンテキストとして読むファイルだ。2024年末から正式にサポートされた。リポジトリにpushしておけば、チームで副業プロジェクトを動かすときも全員が同じ前提で動ける。

## 週次メンテの15分ルール

このシステムの唯一の落とし穴は「更新しないと腐る」ことだ。

私は日曜の夜に15分だけ `CONTEXT.md` を開く。「今週の作業コンテキスト」の欄だけ書き換える。変わった決定事項、詰まったところ、次週の優先タスク。それだけだ。15分で全5ツールの記憶が最新になる。

これを月4回やると60分。副業の生産性維持への投資として、60分は相当に安い。

## 副業への具体的な金銭換算

抽象論にしたくないので数字で書く。

私の副業の実働時間は月40〜50時間程度だ。以前は毎作業セッションの冒頭に平均8〜12分の「コンテキスト説明」が発生していた。月20セッションとすれば200分、約3.3時間。

時給換算で副業の平均単価が3000円/時間なら、毎月約1万円が「AIへの自己紹介」で消えていた計算になる。年間12万円。

`CONTEXT.md` を作るのに最初30分かかった。週次更新が月60分。初年度の維持コストは30分 + 720分 = 約12.5時間。時給3000円換算で37,500円のコストに対して、12万円分の作業時間を取り戻した。

ROIで言えば220%だ。副業における節税と同じ構造で、「払わなくていいコストを特定して削る」という話にすぎない。

## 「全部を覚えさせる」という発想の誤り

最後に一つ言っておく。

`CONTEXT.md` を完璧にしようとすると失敗する。全部書こうとすると200行を超えて、AIが読み切れなくなる。人間も同じで、引き継ぎ資料が100ページあっても誰も読まない。

重要なのは「毎回説明しているのに毎回説明してしまっていること」だけを抽出することだ。

それだけを書く。それだけを更新する。それだけで5つのLLMが同じ前提で動く。

全を動かす操作は、常に1つの動作だ。
